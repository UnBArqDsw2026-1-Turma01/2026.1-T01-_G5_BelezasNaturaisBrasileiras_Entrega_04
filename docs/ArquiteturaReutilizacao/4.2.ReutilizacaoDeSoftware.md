# 4.2. Reutilização de Software

## Introdução

A reutilização de software é um pilar fundamental na engenharia de software moderna, visando aumentar a produtividade, reduzir custos e melhorar a qualidade dos sistemas através do uso de ativos pré-existentes. No projeto **Belezas Naturais Brasileiras (BNB)**, a estratégia de reuso foi aplicada de forma deliberada e sistemática em diversos níveis de abstração.

Este documento detalha como o sistema aproveita componentes externos, estruturas internas e conceitos consolidados para entregar uma solução robusta. A análise segue a taxonomia clássica de reuso (Mili et al., 1995), categorizando as práticas em:
- **Reuso de Caixa-Preta**: Utilização de componentes através de suas interfaces, sem modificação interna.
- **Reuso de Caixa-Branca**: Adaptação e especialização de componentes através de herança e polimorfismo.
- **Reuso de Conceito**: Aplicação de padrões de projeto e padrões arquiteturais documentados.

O objetivo desta documentação é demonstrar que a arquitetura do BNB não foi construída isoladamente, mas sim integrada a um ecossistema de software que potencializa a entrega de valor.

---

## Estratégia de Reuso e Taxonomia

A abordagem de reuso do projeto baseia-se em três níveis de granularidade:

1.  **Nível de Sistema (Caixa-Preta)**: Reuso de frameworks inteiros e bibliotecas de prateleira que fornecem a base infraestrutural.
2.  **Nível de Componente (Caixa-Branca/Interno)**: Reuso de estruturas desenvolvidas pela equipe que são especializadas para atender diferentes requisitos de negócio.
3.  **Nível de Design (Conceito)**: Reuso de soluções arquiteturais e padrões de projeto (GoF) para resolver problemas estruturais recorrentes.

---

## Reutilização Caixa-Preta — Frameworks e Bibliotecas

A reutilização de caixa-preta permite que a equipe se concentre na lógica de negócio (ecoturismo) enquanto delega preocupações transversais para ferramentas especializadas.

### Backend: Justificativa de Escolha

| Componente | Justificativa Arquitetural para o Reuso | Impacto na Qualidade |
| :--- | :--- | :--- |
| **NestJS 11** | Oferece uma estrutura opinativa inspirada em Angular, facilitando o reuso de padrões como DI e Modularização. | **Manutenibilidade**: Padronização da estrutura entre diferentes desenvolvedores. |
| **Prisma 7** | Reuso de um motor de mapeamento objeto-relacional altamente performático e type-safe. | **Confiabilidade**: Redução de erros de SQL em tempo de execução através da tipagem gerada. |
| **Passport** | Reuso de uma infraestrutura de autenticação modular e amplamente testada pela comunidade. | **Segurança**: Mitigação de vulnerabilidades comuns em fluxos de login customizados. |
| **Supabase Auth** | Reuso de um serviço de identidade como plataforma (IDaaS). | **Segurança/Escalabilidade**: Delegar a gestão de senhas e tokens para um provedor especializado. |
| **Class-Validator** | Reuso de lógica de validação baseada em metadados (decorators). | **Produtividade**: Validação declarativa reduz o código boilerplate nos controllers. |

### Frontend: Justificativa de Escolha

| Componente | Justificativa Arquitetural para o Reuso | Impacto na Qualidade |
| :--- | :--- | :--- |
| **React 18** | Reuso de uma biblioteca de UI baseada em componentes, permitindo a criação de bibliotecas internas de design. | **Reusabilidade de UI**: Componentes como botões e cards são reutilizados em todo o SPA. |
| **Tailwind CSS** | Reuso de um sistema de design utility-first que elimina a necessidade de CSS customizado frágil. | **Consistência Visual**: Garante que o espaçamento e as cores sigam uma escala pré-definida. |
| **Vite** | Reuso de um ecossistema de build moderno que acelera o ciclo de desenvolvimento. | **Eficiência**: HMR (Hot Module Replacement) instantâneo melhora a experiência do desenvolvedor. |
| **Axios** | Cliente HTTP robusto com suporte nativo a interceptores, permitindo reuso de lógica de autenticação. | **Modularidade**: Centralização da configuração de chamadas API. |

## Reutilização Caixa-Branca — Especialização e Herança

O reuso de caixa-branca no BNB foca na extensibilidade. Através de classes abstratas e interfaces, criamos "templates" que são preenchidos com lógica específica.

### Padrão Template Method e Herança

O uso de classes abstratas permite o reuso de algoritmos esqueléticos, onde os passos específicos são implementados pelas subclasses.

1.  **Validação de Negócio (Chain of Responsibility)**:
    A classe abstrata `AccountValidationHandler` contém a lógica de encadeamento. As subclasses (Email, Password, Terms) reutilizam o método `setNext` e apenas implementam a regra `handle`. Isso evita a duplicação do código de navegação na lista de validadores.

2.  **Implementações de Repositório**:
    Embora cada repositório seja único, eles implementam interfaces de domínio (ex: `ITrilhaRepository`). O reuso aqui é de **contrato**. O Use Case reutiliza o mesmo contrato, independentemente se a implementação por trás é `PrismaTrilhaRepository` ou um `MockTrilhaRepository` para testes.

3.  **Adaptadores de Comunicação**:
    O `IAuthAdapter` permite que diferentes estratégias de autenticação (Local, Google, OAuth) compartilhem o mesmo contrato esperado pelo sistema, permitindo que o `AuthService` seja reutilizado sem modificações para diferentes provedores.

## Reutilização de Conceito — Padrões GoF

A reutilização de conceito é a forma mais refinada de reuso, pois não reaproveita código, mas sim a **experiência e inteligência de design**. No BNB, implementamos intencionalmente uma vasta gama de padrões GoF para garantir que o sistema siga princípios de design sólido (SOLID).

### Matriz de Benefício de Reuso por Padrão

| Categoria | Padrão | Benefício de Reuso / Desacoplamento |
| :--- | :--- | :--- |
| **Criacional** | **Builder** | Reutiliza a lógica de construção de objetos complexos, permitindo diferentes representações (ex: DTOs simplificados vs completos). |
| | **Object Pool** | Reutiliza instâncias custosas (conexões de chat), evitando o overhead de alocação/desalocação constante. |
| **Estrutural** | **Proxy** | Adiciona funcionalidades (Cache, Log, Auth) sem tocar na lógica do componente original, promovendo o reuso do componente base em diferentes contextos (ex: com ou sem cache). |
| | **Adapter** | Permite que o sistema reuse bibliotecas de terceiros cujas interfaces são incompatíveis com o domínio interno. |
| **Comportamental** | **Mediator** | Promove o reuso de componentes de negócio ao centralizar suas comunicações, evitando o acoplamento "espaguete" entre classes. |
| | **Observer** | Permite adicionar novos "interessados" em eventos do sistema (ex: novas conquistas) sem alterar o disparador original. |

## Reutilização Interna — Bibliotecas de Domínio e Infra

O sistema BNB foi projetado para que seus próprios módulos sirvam como fonte de reuso interno.

### Componentes Compartilhados (Shared Kernel)

Localizados em `backend/src/shared`, estes componentes são os "cidadãos de primeira classe" do reuso no projeto:

- **Infraestrutura de Segurança**: Os `Guards` de autenticação e autorização são aplicados de forma declarativa em todos os módulos. O código de validação de JWT é escrito uma vez e reutilizado em 100% dos endpoints protegidos.
- **Utilitários de Banco de Dados**: O `PrismaService` é um Singleton injetado globalmente. Isso reutiliza a pool de conexões do banco em todos os módulos verticais, otimizando recursos do sistema.
- **Mappers de Dados**: Os `Mappers` (`UserMapper`, `TrilhaMapper`) são reutilizados tanto na camada de aplicação (para devolver DTOs) quanto na camada de infraestrutura (para converter modelos do banco), garantindo uma única fonte de verdade para a transformação de dados.

## Comprobatório Prático de Execução: Reutilização de Componentes React

Conforme diretriz específica de entrega ("*Se Reutilização, mostrar código e execução do caso de reutilização sendo esse o uso de Components do React*"), destacamos a componentização estratégica no frontend. 

O sistema reaproveita o componente `<ProtectedRoute />` para centralizar as regras de bloqueio de navegação, validando autenticação e checando Autorização por Papéis (RBAC).

**1. O Componente Reutilizado (`ProtectedRoute.tsx`):**
```tsx
import { Navigate } from 'react-router-dom'
import { useAuth } from '../contexts/AuthContext'

export default function ProtectedRoute({ children, roles }) {
  const { isAuthenticated, user } = useAuth()

  // Regra central de bloqueio de acesso
  if (!isAuthenticated) return <Navigate to="/login" replace />
  if (roles && user && !roles.includes(user.role)) return <Navigate to="/" replace />

  // Caso cumpra as regras, renderiza a tela filha
  return <>{children}</>
}
```

**2. Reuso e Aplicação (em `App.tsx`):**
O componente é invocado em múltiplas definições de rotas. Dessa forma, as telas em si (`Chat`, `CriarTrilha`, `PainelAdmin`) não precisam repetir código para checar se o usuário possui permissão.

```tsx
{/* Reuso nível 1: Exige apenas usuário logado */}
<Route path="/chat" element={
  <ProtectedRoute>
    <Layout><Chat /></Layout>
  </ProtectedRoute>
} />

{/* Reuso nível 2: Exige cargos específicos (Organizadores e Admins) */}
<Route path="/trilhas/criar" element={
  <ProtectedRoute roles={['ORGANIZER', 'ADMIN']}>
    <Layout><CriarTrilha /></Layout>
  </ProtectedRoute>
} />

{/* Reuso nível 3: Acesso Estrito (Apenas Admin) */}
<Route path="/admin" element={
  <ProtectedRoute roles={['ADMIN']}>
    <Layout><PainelAdmin /></Layout>
  </ProtectedRoute>
} />
```

## Análise de Impacto e Métricas de Reuso

Embora o projeto não utilize métricas quantitativas formais (como LOC de reuso), uma análise qualitativa revela:

1.  **Velocidade de Desenvolvimento**: O uso de NestJS e React permitiu a montagem do "esqueleto" do sistema em dias, o que levaria semanas se construído sobre bibliotecas de baixo nível.
2.  **Redução de Bugs**: Ao reutilizar o `class-validator`, eliminamos uma classe inteira de bugs relacionados a entradas malformadas, pois a lógica de validação é proveniente de uma biblioteca exaustivamente testada.
3.  **Consistência**: O reuso do padrão Proxy para cache garante que a estratégia de performance seja idêntica em diferentes módulos, facilitando o diagnóstico de problemas.

## Senso Crítico e Trade-offs de Reuso

### O Custo da Abstração
A reutilização extrema através de padrões GoF em um sistema pequeno introduz camadas de "indireção". Um desenvolvedor novo precisa navegar por interfaces, proxies e adaptadores antes de chegar à lógica de persistência. Este é o **custo do reuso**: a complexidade cognitiva aumenta em troca de uma maior flexibilidade futura.

### Acoplamento com o Framework
Ao reutilizar o NestJS de forma profunda (decorators, modules, DI), o sistema torna-se dependente dele. Uma migração para outro framework (como Fastify puro ou Go) exigiria uma reescrita significativa. O reuso de caixa-preta gera um **lock-in tecnológico**.

### Reuso vs Duplicação (Regra de Três)
Em alguns pontos, optamos pela duplicação em vez do reuso prematuro. Por exemplo, os Mappers de cada módulo são independentes. Embora compartilhem a mesma estrutura, o reuso de um "GenericMapper" poderia criar um acoplamento indesejado entre domínios que evoluem em ritmos diferentes.

---

## Conclusão

A estratégia de reutilização de software no **Belezas Naturais Brasileiras** é um exemplo de equilíbrio entre o uso pragmático de ferramentas de mercado (Caixa-Preta) e o investimento em uma arquitetura extensível baseada em padrões (Caixa-Branca e Conceito). O reuso não foi visto apenas como "copiar código", mas como uma forma de garantir que o sistema seja construído sobre bases sólidas, prontas para evolução e manutenção a longo prazo.
