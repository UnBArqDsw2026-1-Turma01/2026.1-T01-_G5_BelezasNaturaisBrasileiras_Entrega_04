# 4.2. Reutilização de Software

## Introdução

Este documento registra as práticas de reutilização de software observadas no projeto **Belezas Naturais Brasileiras**, organizadas nas três categorias clássicas: caixa-preta (uso de componentes sem conhecer a implementação), caixa-branca (herança e especialização de componentes existentes) e reutilização de conceito (aplicação de padrões).

## Reutilização Caixa-Branca — Especialização e Herança

Componentes do projeto que herdam de classes abstratas ou implementam interfaces para especialização:

### Chain of Responsibility — Validação de Cadastro

```
AccountValidationHandler (abstract)
  ├── EmailUniquenessHandler
  ├── PasswordStrengthHandler
  └── TermsAcceptanceHandler
```

`AccountValidationHandler` define o contrato `handle(input)` e o encadeamento `setNext(handler)`. Cada handler concreto herda e implementa apenas a regra de validação específica, reutilizando o mecanismo de passagem para o próximo handler.

### Adapter — Integrações Externas

```
IAuthAdapter (interface)
  ├── GoogleAuthAdapter
  └── LocalAuthAdapter

IMapAdapter (interface)
  └── GoogleMapsAdapter

INotificationAdapter (interface)
  ├── TwilioAdapter
  └── LoggerNotificationChannel
```

As interfaces definem os contratos. Cada implementação concreta encapsula um SDK diferente. O `AuthAdapterService`, `MapAdapterService` e `NotificationAdapterService` dependem apenas das interfaces — trocar a implementação não exige alterar o serviço.

### Factory — Criação de Usuários por Role

```
IUserFactory (interface)
  ├── AdminUserFactory
  ├── CommonUserFactory
  └── OrganizerUserFactory
```

`UserFactoryRegistry` mapeia `UserRole → IUserFactory`. O `CreateAccountUseCase` consulta o registry e delega a criação à factory correta sem saber qual factory é usada.

### Strategy — Ordenação de Trilhas

```
ITrilhaOrdenacaoStrategy (interface)
  ├── OrdenarTrilhasPorDataStrategy
  └── OrdenarTrilhasPorTituloStrategy
```

`TrilhaOrdenacaoContext` recebe a estratégia via injeção; o `ListarTrilhasUseCase` não precisa conhecer o algoritmo de ordenação.

### Mediator — Lifecycle Handlers

```
ILifecycleHandler (interface)
  ├── TrailStateHandler
  ├── AttendanceHandler
  ├── BadgeHandler
  └── HistoryNotificationHandler
```

Cada handler implementa a interface e encapsula um efeito colateral do fluxo de finalização de trilha. O `TrailLifecycleMediatorService` os chama em sequência sem que os handlers se conheçam.

## Reutilização de Conceito — Padrões GoF

Os padrões GoF são reutilizações de soluções documentadas para problemas recorrentes (Gamma et al., 1994). A tabela abaixo registra os 20+ padrões aplicados e o problema que cada um resolve no projeto:

| Padrão                  | Problema resolvido no projeto                                                                                         |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------- |
| Singleton               | `ConfirmationCodeService` precisa de estado compartilhado (mapa de códigos ativos) sem duplicação entre instâncias    |
| Builder                 | Construção de `User` envolve validação e campos opcionais; evita construtores com muitos parâmetros                   |
| Factory Method          | Diferentes roles produzem objetos `User` com comportamentos distintos; factory centraliza essa lógica                 |
| Factory Registry        | O `CreateAccountUseCase` não deve conhecer as factories concretas; o registry resolve pelo role                       |
| Prototype               | `PromoteUserUseCase` precisa criar um `User` com role diferente sem repetir todos os campos; `User.clone()` resolve   |
| Object Pool             | Criação de conexões de chat é custosa; o pool reutiliza até 50 conexões                                               |
| Adapter                 | SDKs externos (`Twilio`, `Google Maps`, `Supabase`) têm APIs incompatíveis com o domínio; adapters normalizam         |
| Composite               | Localizações formam uma hierarquia (estado → cidade → pontos) percorrida recursivamente                               |
| Decorator               | `AuditedTrilhaRepository` adiciona auditoria a qualquer repositório sem alterar a implementação original              |
| Facade                  | Controllers delegam a uma única interface (`TrilhaFacade`, `InscricaoFacade`) em vez de coordenar múltiplos use cases |
| Flyweight               | `LocalizacaoFlyweightFactory` compartilha instâncias de localização com mesmo estado intrinsic                        |
| Proxy (Protection)      | Operações de escrita em trilhas e pontos devem verificar propriedade antes de executar                                |
| Proxy (Cache)           | Leituras repetidas ao banco são evitadas com cache em memória                                                         |
| Chain of Responsibility | Validações de signup são independentes e extensíveis; cada handler cuida de uma regra                                 |
| Command                 | Edições de trilha precisam de histórico e capacidade de undo                                                          |
| Iterator                | `ListarTrilhasUseCase` precisa filtrar e paginar sem expor a coleção interna                                          |
| Mediator                | Finalizar trilha dispara múltiplos efeitos; o mediator desacopla os handlers entre si                                 |
| Memento                 | Estado de trilha deve ser salvo antes de edições destrutivas para permitir rollback                                   |
| Observer                | Finalização de trilha dispara notificações e distribuição de badges sem acoplamento no use case                       |
| Strategy                | Ordenação de trilhas muda por parâmetro de requisição; strategies são trocadas sem alterar o use case                 |
| Visitor                 | Percorre inscrições para aplicar operações (distribuição de badge, notificação) sem modificar `Inscricao`             |

## Reutilização Interna — Componentes do Projeto

Soluções construídas pela equipe que são reutilizadas em mais de um módulo:

| Componente                                                     | Onde é reutilizado                                                                                                                                                |
| -------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `JwtAuthGuard` + `RolesGuard`                                  | Todos os controllers com rotas protegidas (`accounts`, `trilhas`, `inscricoes`, `pontos-turisticos`, `chat`)                                                      |
| `PrismaService`                                                | Todos os repositórios de infraestrutura (`PrismaUserRepository`, `PrismaTrilhaRepository`, `ChatSessionRepository`...)                                            |
| Pattern Proxy (Protection + Cache)                             | Dois módulos independentes: `trilhas` (via `TrilhaProxyRepository` + `CachedTrilhaRepository`) e `pontos-turisticos` (via `PontosAuthProxy` + `PontosCacheProxy`) |
| `TrilhaMapper`, `InscricaoMapper`, `UserMapper`, `BadgeMapper` | Conversão bidirecional entre objeto Prisma e entidade de domínio em todos os repositórios Prisma                                                                  |
| Interface `IObjectPool<T>`                                     | Implementada por `ChatObjectPoolService`; interface genérica reutilizável para qualquer tipo de pool                                                              |

## Exemplo com Código — A Mesma Solução em Dois Módulos

O conceito de Cache Proxy foi reaplicado em dois módulos independentes. Os trechos abaixo são do código real:

**Módulo trilhas — cache por entidade e por listagem, invalidado na escrita:**

> [`backend/src/modules/trilhas/infrastructure/persistence/CachedTrilhaRepository.ts`](https://github.com/UnBArqDsw2026-1-Turma01/2026.1-T01-_G5_BelezasNaturaisBrasileiras_Entrega_04/blob/main/backend/src/modules/trilhas/infrastructure/persistence/CachedTrilhaRepository.ts)

```typescript
export class CachedTrilhaRepository implements ITrilhaRepository {
  private readonly cacheById = new Map<string, Trilha>();
  private cacheAll: Trilha[] | null = null;

  constructor(private readonly wrapped: ITrilhaRepository) {}

  async findById(id: string): Promise<Trilha | null> {
    if (this.cacheById.has(id)) {
      return this.cacheById.get(id)!;
    }
    const trilha = await this.wrapped.findById(id);
    if (trilha) {
      this.cacheById.set(id, trilha);
    }
    return trilha;
  }

  async save(trilha: Trilha): Promise<Trilha> {
    const saved = await this.wrapped.save(trilha);
    this.cacheById.delete(trilha.id);
    this.cacheAll = null;
    return saved;
  }
  // ...
}
```

**Módulo pontos-turisticos — cache por chave de filtro com TTL de 3 minutos:**

> [`backend/src/modules/pontos-turisticos/interface/proxies/PontosCacheProxy.ts`](https://github.com/UnBArqDsw2026-1-Turma01/2026.1-T01-_G5_BelezasNaturaisBrasileiras_Entrega_04/blob/main/backend/src/modules/pontos-turisticos/interface/proxies/PontosCacheProxy.ts)

```typescript
@Injectable()
export class PontosCacheProxy implements IPontosTuristicosService {
  private readonly cache = new Map<
    string,
    { dados: any[]; expiraEm: number }
  >();
  private readonly TTL_MS = 3 * 60 * 1000; // 3 minutos

  constructor(private readonly proximo: IPontosTuristicosService) {}

  async buscarFeed(filtros: Record<string, any>) {
    const chave = JSON.stringify(filtros || {});
    const entrada = this.cache.get(chave);

    if (entrada && Date.now() < entrada.expiraEm) {
      this.logger.log(`Cache HIT: ${chave}`);
      return entrada.dados;
    }

    const dados = await this.proximo.buscarFeed(filtros);
    this.cache.set(chave, { dados, expiraEm: Date.now() + this.TTL_MS });
    return dados;
  }
  // ...
}
```

As duas implementações compartilham a estrutura (implementam a mesma interface do componente real, recebem o próximo elo no construtor, interceptam leituras), mas divergem na política: o cache de trilhas invalida na escrita e não expira; o de pontos expira por TTL e limpa tudo em qualquer escrita. A duplicação de estrutura com políticas diferentes é o argumento para a generalização discutida no Senso Crítico.

## Senso Crítico

**Frameworks como reutilização vs acoplamento:** o uso de NestJS impõe o uso de decorators TypeScript, o que cria acoplamento entre o código de domínio e anotações do framework quando usadas fora da camada `interface`. No projeto, `@Injectable()` aparece em classes de `infrastructure` — aceitável. Se aparecer em `domain`, viola o isolamento.

**20+ padrões num sistema pequeno:** a aplicação de todos os 23 padrões GoF (ou próximo disso) num CRUD com menos de 15 endpoints é over-engineering por definição. A justificativa do projeto é acadêmica (demonstrar os padrões), não orgânica. Em um produto real, a maioria dessas abstrações seria desnecessária e aumentaria o custo de manutenção.

**Proxy cache duplicado:** o mesmo problema (cache de leituras) foi resolvido duas vezes com implementações separadas (`CachedTrilhaRepository` e `PontosCacheProxy`). A solução genérica (`IObjectPool<T>`) sugere que uma classe `CacheProxy<T>` genérica poderia ter sido criada e reutilizada, reduzindo duplicação.

**Ausência de testes no frontend:** React, react-router-dom, axios e Context API são reutilizados, mas sem cobertura de testes. A reutilização de código sem testes cria risco quando há atualização de versão ou mudança de comportamento.
