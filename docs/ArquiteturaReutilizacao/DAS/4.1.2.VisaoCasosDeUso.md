# 4.1.2. Visão de Casos de Uso

## Introdução

A Visão de Casos de Uso, no modelo 4+1 de Kruchten, não descreve todos os casos de uso do sistema — ela seleciona os **casos de uso arquiteturalmente significativos**: aqueles que motivaram decisões de estrutura, impuseram restrições de design ou resultaram na adoção de padrões específicos.

---

## Diagrama de Casos de Uso

O diagrama abaixo foi produzido na Fase 2 (Modelagem Organizacional) e representa os casos de uso do sistema com os três atores principais:

![Diagrama de Casos de Uso — V3](../../assets/fase_2/images/Diagrama%20OrganizacionalV3.drawio.png)

> Fonte: [2.3. Modelagem Organizacional e Casos de Uso](../../Modelagem/2.3.ModelagemOrganizacionalCasosDeUso.md) — Fase 2.

**Atores:**

- **Usuário Comum (COMMON_USER):** cadastra-se, explora pontos turísticos e trilhas, se inscreve em trilhas, faz check-in, recebe badges, usa chat
- **Organizador (ORGANIZER):** cria e edita trilhas, aceita/rejeita inscrições, gera código de confirmação, finaliza trilhas
- **Administrador (ADMIN):** promove usuários para ORGANIZER, acessa painel administrativo

## Casos de Uso Arquiteturalmente Significativos

Os seis casos de uso abaixo tiveram impacto direto nas decisões arquiteturais documentadas nas demais visões:

### UC01 — Cadastro de Conta (Signup)

**Fluxo:** `POST /accounts/signup`

**Impacto arquitetural:**

- Motivou a **Chain of Responsibility** (`EmailUniquenessHandler → PasswordStrengthHandler → TermsAcceptanceHandler`) para validação extensível sem `if/else` encadeado
- Motivou o **Factory Method + Factory Registry** para criar o tipo de `User` correto por role, sem acoplamento no use case
- Obrigou a separação entre Supabase Auth (credenciais) e Prisma (perfil), com `UserMapper` como ponto de conversão

**Decisão arquitetural gerada:** autenticação delegada ao Supabase; backend emite JWT próprio (HS256) com `JWT_SECRET`

---

### UC02 — Criar e Editar Trilha (Organizer)

**Fluxo:** `POST /trilhas`, `PATCH /trilhas/:id`

**Impacto arquitetural:**

- Motivou o **Command + TrilhaCommandHistory** para encapsular edições e permitir undo
- Motivou o **Memento + TrilhaCaretaker** para salvar o estado antes de cada edição
- Motivou o **TrilhaRequestContext** (AsyncLocalStorage) para propagar o `organizadorId` até o proxy de proteção sem passar pelo call stack
- Motivou o **TrilhaProxyRepository** (Protection Proxy) para verificar propriedade da trilha antes do `save()`

**Decisão arquitetural gerada:** o contexto de execução (quem está fazendo a operação) é armazenado em `AsyncLocalStorage`, não passado explicitamente como parâmetro nos repositórios

---

### UC03 — Listar Trilhas (Filtro + Paginação)

**Fluxo:** `GET /trilhas`

**Impacto arquitetural:**

- Motivou os **Iterators** (`TrilhaFilteredIterator`, `TrilhaPaginatedIterator`) para separar a lógica de filtro e paginação da lógica de acesso ao repositório
- Motivou a **Strategy** (`OrdenarTrilhasPorDataStrategy`, `OrdenarTrilhasPorTituloStrategy`) para ordenação intercambiável sem modificar o use case

**Decisão arquitetural gerada:** `ListarTrilhasUseCase` não recebe listas prontas do repositório — recebe um `ITrilhaIterator` e percorre de forma controlada

---

### UC04 — Finalizar Trilha (Organizer)

**Fluxo:** `POST /trilhas/:id/finalizar`

**Impacto arquitetural:**

- É o caso de uso com mais efeitos colaterais: muda status da trilha, valida presenças, distribui badges, grava histórico, notifica participantes
- Motivou o **Mediator** (`TrailLifecycleMediatorService` com 4 handlers) para desacoplar os efeitos colaterais entre si
- Motivou o **Visitor** (`BadgeDistribuicaoVisitor`, `NotificacaoInscricaoVisitor`) para aplicar badges e notificações por inscrição, despachando por status
- Motivou o **Observer** (`TrilhaEventEmitter` com `BadgeDistribuicaoObserver` e `NotificacaoObserver`) como infraestrutura de eventos — inscrito no init do módulo, exercitado em testes
- Motivou o **Memento** (salva estado antes de finalizar, permitindo rollback via `RestaurarTrilhaUseCase`)

**Decisão arquitetural gerada:** a sequência Memento → Mediator → Visitor é executada nessa ordem dentro do `FinalizarTrilhaUseCase`, com cada padrão responsável por uma parte dos efeitos

---

### UC05 — Chat entre Usuário e Organizador

**Fluxo:** `POST /chat/sessions`, `POST /chat/sessions/:id/messages`

**Impacto arquitetural:**

- Motivou o **Object Pool** (`ChatObjectPoolService`) para reuso de conexões sem overhead de criação/destruição a cada mensagem; o pool retém até 50 conexões ociosas no `release()`
- Motivou o **ChatConnectionFactoryService** para criar novas conexões quando o pool está vazio ou a conexão retirada não está mais viva

**Decisão arquitetural gerada:** chat é stateful no processo NestJS (pool em memória); não há WebSocket — as mensagens são salvas em `ChatActivity` no banco

---

### UC06 — Integração com Serviços Externos (Maps, SMS, OAuth)

**Fluxo:** `POST /adapters/geocode`, `POST /adapters/notify/sms`, `POST /adapters/auth/validate`

**Impacto arquitetural:**

- Motivou o **Adapter** com interfaces comuns (`IMapAdapter`, `INotificationAdapter`, `IAuthAdapter`) para isolar o sistema dos SDKs externos
- A troca de provider (ex.: Google Maps → OpenStreetMap) exige apenas uma nova implementação da interface, sem alterar o `MapAdapterService`

**Decisão arquitetural gerada:** nenhum controller ou use case importa SDK externo diretamente; todos passam pelo adapter service correspondente

## Senso Crítico

A seleção de casos de uso arquiteturalmente significativos é subjetiva. Os critérios usados aqui foram: (1) o caso de uso gerou um padrão ou componente que não existiria sem ele; (2) o caso de uso impôs uma restrição de design que afeta outros módulos; (3) o caso de uso representa um risco técnico (concorrência, integração externa, segurança).

Casos de uso como "visualizar perfil" ou "criar ponto turístico" não aparecem aqui porque, embora façam parte do produto, não geraram decisões arquiteturais fora do padrão CRUD sobre Prisma.

A manutenção deste documento depende de atualizar a seleção quando novos casos de uso mudarem a arquitetura — o que não ocorreu entre as Fases 2 e 4 para os itens listados.
