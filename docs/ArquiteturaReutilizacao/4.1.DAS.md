# 4.1. Documento de Arquitetura de Software (DAS)

## Introdução

Este documento descreve a arquitetura do sistema **Belezas Naturais Brasileiras**, uma plataforma para descoberta e organização de ecoturismo no Brasil. O DAS cobre o backend (NestJS 11 + Prisma 7 + PostgreSQL 15) e o frontend (React 18 + Vite 5), organizados em monorepo.

A estrutura segue o modelo de visões **4+1** (Kruchten, 1995), com adição de uma Visão de Dados.

---

## Escopo

O documento cobre:

- A organização interna do backend em módulos verticais e camadas
- As interfaces e contratos entre componentes
- Os fluxos de execução dos principais casos de uso
- A topologia de implantação local e as dependências externas
- O modelo de dados relacional (PostgreSQL via Prisma)

Fora do escopo: especificação de SLAs, infraestrutura de produção (não definida no projeto), plano de escalabilidade.

---

## Representação Arquitetural

O sistema utiliza os seguintes estilos e padrões:

| Estilo / Padrão                         | Aplicação                                                                                                           |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| **Monólito modular**                    | Backend como único processo deployável, internamente dividido em módulos verticais isolados                         |
| **Arquitetura em camadas (DDD tático)** | Cada módulo segue as camadas `domain → application → infrastructure → interface` com regras de dependência estritas |
| **REST**                                | API HTTP com verbos e status codes padrão; autenticação via JWT (Bearer)                                            |
| **SPA (Single-Page Application)**       | Frontend React 18 consumindo a API REST via axios                                                                   |
| **Padrões GoF**                         | 20+ padrões implementados nas camadas de domínio e infraestrutura (ver Visão Lógica)                                |

---

## Metas e Restrições Arquiteturais

| Meta / Restrição          | Descrição                                                                                                                 |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| Isolamento do domínio     | `domain` não importa nenhuma classe de `infrastructure` ou `interface`; Prisma nunca cruza a barreira de `infrastructure` |
| Inversão de dependência   | Use cases dependem de interfaces (`IUserRepository`, `ITrilhaRepository`...), não de implementações concretas             |
| Autenticação centralizada | Supabase Auth é a única fonte de verdade para credenciais; senhas nunca persistem no PostgreSQL local                     |
| Autenticação JWT (HS256)  | Backend emite JWT assinado com `JWT_SECRET`; `JwtAuthGuard` valida em rotas protegidas                                    |
| RBAC                      | Três roles: `COMMON_USER`, `ORGANIZER`, `ADMIN`; controlado via `RolesGuard` + `@Roles()`                                 |
| PostgreSQL único          | Prisma é o único ORM; sem suporte a múltiplos bancos                                                                      |
| Build único por processo  | Frontend e backend são processos separados (Vite dev server / NestJS); não há SSR                                         |

## Roteiro de Demonstração (Execução)

A entrega exige mostrar o sistema rodando, evidenciando padrões e estilos arquiteturais. Roteiro:

```bash
# 1. Infraestrutura (PostgreSQL + pgAdmin via Docker)
cd backend
docker-compose up -d

# 2. Migrations e API
npm run prisma:migrate:dev
npm run start:dev            # API em :3000

# 3. Frontend (outro terminal)
cd ../frontend && npm run dev   # SPA em :5173
```

**Sequência de demonstração via API** (evidencia os padrões em execução):

| Passo | Requisição                                              | O que evidencia                                                  |
| ----- | ------------------------------------------------------- | ---------------------------------------------------------------- |
| 1     | `POST /accounts/signup` com senha fraca                 | Chain of Responsibility rejeitando (`PasswordStrengthHandler`)   |
| 2     | `POST /accounts/signup` válido + `POST /accounts/login` | Factory por role + JWT HS256                                     |
| 3     | `POST /trilhas` sem token / com token COMMON_USER       | Guards: 401 / 403 (estilo em camadas + RBAC)                     |
| 4     | `POST /trilhas` com token ORGANIZER                     | Facade → Use Case → cadeia de proxies                            |
| 5     | `GET /trilhas` duas vezes seguidas                      | Cache Proxy (segunda chamada sem query ao banco — observar logs) |
| 6     | `POST /inscricoes/trilha/:id` → aceitar → check-in      | Singleton de códigos de confirmação                              |
| 7     | `POST /trilhas/:id/finalizar`                           | Memento + Mediator (4 handlers) + Visitor                        |
| 8     | `GET /trilhas/badges/minhas`                            | Badge distribuído pela finalização                               |
| 9     | `GET /chat/pool/status` antes/depois de mensagens       | Object Pool (`poolSize`)                                         |
| 10    | `cd backend && npm run test`                            | Specs unitários dos padrões passando                             |

Requisições prontas em [`CURLS_MANUAL.md`](https://github.com/UnBArqDsw2026-1-Turma01/2026.1-T01-_G5_BelezasNaturaisBrasileiras_Entrega_04/blob/main/CURLS_MANUAL.md) e na coleção Postman [`BNB_Collection.postman_collection.json`](https://github.com/UnBArqDsw2026-1-Turma01/2026.1-T01-_G5_BelezasNaturaisBrasileiras_Entrega_04/blob/main/BNB_Collection.postman_collection.json).

## Senso Crítico

**Por que 4+1 e não outro modelo:** o C4 Model (Brown) seria uma alternativa mais recente, com níveis de zoom (contexto, contêiner, componente, código). O 4+1 foi escolhido por ser o modelo de referência da disciplina e por separar preocupações por audiência — o que distribui melhor o trabalho entre os membros da equipe. O custo é a sobreposição parcial entre visões (a Visão Lógica e a de Implementação descrevem estruturas próximas em níveis diferentes).

**DAS escrito após o código:** este DAS foi produzido por engenharia reversa do código existente, não antes dele. Isso inverte o papel clássico do documento (guiar a implementação), mas garante fidelidade: cada afirmação tem rastro a um arquivo do repositório. O risco é o documento desatualizar conforme o código evolui — sem processo de revisão contínua, as visões refletem o estado de junho/2026.

**Limites do escopo:** as visões não cobrem requisitos não-funcionais quantificados (latência, throughput, disponibilidade) porque o projeto não os definiu. Um DAS de produção precisaria dessas metas para justificar decisões de escala.
