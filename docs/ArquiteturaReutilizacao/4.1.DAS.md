# 4.1. Documento de Arquitetura de Software (DAS)

## Introdução

Este documento apresenta uma visão abrangente e detalhada da arquitetura do sistema **Belezas Naturais Brasileiras (BNB)**. O objetivo é descrever as decisões arquiteturais fundamentais que orientam o desenvolvimento, garantindo que o sistema seja robusto, escalável, manutenível e atenda aos requisitos de negócio para a promoção do ecoturismo no Brasil.

O DAS atua como um guia para desenvolvedores, arquitetos e stakeholders, facilitando a compreensão da estrutura interna, das interações entre componentes e das tecnologias escolhidas. A arquitetura aqui descrita reflete o estado atual do sistema na entrega da Fase 4, consolidando as práticas de reuso e padrões de projeto aplicados.

---

## Contexto e Escopo

### Contexto do Sistema
O "Belezas Naturais Brasileiras" é uma plataforma digital que visa conectar entusiastas do ecoturismo a trilhas, pontos turísticos e experiências naturais em todo o território nacional. O sistema permite a descoberta de novos destinos, a inscrição em trilhas organizadas e a gamificação da experiência através de badges e check-ins.

### Escopo Arquitetural
Este documento cobre:
- **Estrutura do Backend**: Organização em NestJS, módulos verticais e camadas DDD.
- **Estrutura do Frontend**: Aplicação SPA em React e comunicação com a API.
- **Modelo de Persistência**: Esquema relacional via Prisma ORM e PostgreSQL.
- **Segurança**: Fluxos de autenticação JWT e autorização RBAC.
- **Padrões de Projeto**: Implementação de mais de 20 padrões GoF (Criacionais, Estruturais e Comportamentais).
- **Topologia de Implantação**: Configuração de contêineres e serviços externos.

---

## Representação Arquitetural

A arquitetura do BNB é baseada no modelo de visões **4+1** de Philippe Kruchten, complementada por uma **Visão de Dados**. Esta abordagem permite analisar o sistema sob diferentes perspectivas de stakeholders.

### Estilos e Padrões Fundamentais

| Estilo / Padrão | Descrição e Justificativa |
| :--- | :--- |
| **Monólito Modular** | O backend é implantado como uma unidade única, mas organizado internamente em módulos verticais independentes. Isso facilita o desenvolvimento inicial e permite uma futura transição para microsserviços, se necessário. |
| **Arquitetura em Camadas (DDD Tático)** | Cada módulo segue as camadas `domain`, `application`, `infrastructure` e `interface`. Esta separação garante que as regras de negócio (domínio) fiquem isoladas de detalhes técnicos como bancos de dados ou frameworks. |
| **API RESTful** | Comunicação baseada em recursos HTTP, utilizando verbos padrão (GET, POST, PUT, DELETE) e códigos de status semânticos, facilitando a integração com diversos clientes. |
| **SPA (Single-Page Application)** | Frontend React que carrega uma única página e atualiza dinamicamente o conteúdo, proporcionando uma experiência de usuário fluida similar a aplicativos desktop. |
| **Injeção de Dependência** | Utilização extensiva do container de IoC (Inversion of Control) do NestJS para gerenciar o ciclo de vida e a desacoplagem de componentes. |

## Decisões Arquiteturais Significativas (SADs)

| Decisão | Motivação | Impacto |
| :--- | :--- | :--- |
| **Uso de NestJS** | Framework Node.js opinativo que fornece uma estrutura sólida e segue princípios de SOLID nativamente. | Curva de aprendizado inicial, mas alta manutenibilidade e testabilidade a longo prazo. |
| **Prisma ORM** | Type-safety ponta a ponta e facilidade em migrações de banco de dados. | Desenvolvimento acelerado do modelo de dados e redução de erros em tempo de execução. |
| **Supabase Auth** | Terceirização da gestão de credenciais e segurança de identidade. | Redução de riscos de segurança (senhas nunca tocam nosso banco) e facilidade na implementação de OAuth. |
| **Cadeia de Proxies para Repositórios** | Necessidade de adicionar funcionalidades transversais (cache, auditoria, proteção) sem poluir o código do repositório base. | Implementação limpa do padrão Proxy e Decorator, permitindo ativar/desativar funcionalidades via configuração do módulo. |

## Metas e Restrições Arquiteturais

### Metas
- **Modularidade**: Módulos devem ser isolados; a comunicação entre eles deve ocorrer via interfaces ou eventos (Mediator).
- **Testabilidade**: O código deve permitir a execução de testes unitários rápidos, especialmente na camada de domínio, sem necessidade de banco de dados.
- **Segurança**: Proteção de rotas sensíveis e garantia de que usuários só acessem dados permitidos pelo seu papel (RBAC).

### Restrições
- **Linguagem**: TypeScript deve ser utilizado em todo o stack (Frontend e Backend) para garantir consistência de tipos.
- **Banco de Dados**: Uso obrigatório de PostgreSQL 15+ para persistência relacional.
- **Autenticação**: O backend deve ser stateless, utilizando JWT para identificação do usuário.

## Padrões de Projeto Aplicados (GoF)

O BNB utiliza uma ampla gama de padrões de projeto para resolver problemas recorrentes de design e promover o reuso de código. A implementação segue as três categorias clássicas:

### 1. Padrões Criacionais
- **Factory Method**: Utilizado na criação de instâncias de usuários e trilhas, encapsulando a lógica de inicialização complexa.
- **Singleton**: Aplicado ao `PrismaService` e gerenciadores de configuração, garantindo uma única instância de conexão com o banco de dados.
- **Builder**: Facilita a construção de objetos complexos em testes e na montagem de DTOs de resposta.
- **Object Pool**: Gerencia instâncias de conexões de chat, otimizando o uso de recursos de rede.

### 2. Padrões Estruturais
- **Proxy**: Utilizado para controle de acesso (Protection Proxy), cache de resultados (Cache Proxy) e auditoria (Logging Decorator) nos repositórios.
- **Facade**: Camada de aplicação que simplifica o acesso às funcionalidades dos módulos, escondendo a complexidade das interações entre entidades.
- **Adapter**: Padroniza a interface de serviços externos (como Supabase Auth ou adaptadores de mapas), permitindo a troca de provedores sem alterar o domínio.
- **Composite**: Usado para representar estruturas de pontos turísticos que podem conter outros sub-pontos.

### 3. Padrões Comportamentais
- **Chain of Responsibility**: Aplicado na validação de força de senha e sanitização de entradas durante o signup.
- **Mediator**: Centraliza a comunicação entre componentes de chat, evitando dependências diretas entre remetentes e destinatários.
- **Observer**: Notifica módulos de gamificação quando uma trilha é finalizada para atribuição de badges.
- **Memento**: Permite salvar e restaurar o estado de uma trilha durante processos de edição complexos.
- **Visitor**: Utilizado para percorrer a estrutura de inscritos em uma trilha e gerar relatórios ou realizar check-ins em lote.

## Atributos de Qualidade

A arquitetura do BNB prioriza os seguintes atributos de qualidade (baseados na ISO/IEC 25010):

1. **Manutenibilidade**: A divisão em camadas e o uso de DI facilitam a substituição de componentes (ex: trocar Prisma por outro ORM) com impacto mínimo.
2. **Segurança**: Implementada através de `Guards` do NestJS, validando JWTs e permissões em nível de rota e de serviço (Proxy de Proteção).
3. **Usabilidade**: O frontend React com Vite garante tempos de carregamento rápidos e uma interface reativa.
4. **Portabilidade**: O uso de Docker para o backend e banco de dados garante que o sistema possa ser executado em qualquer ambiente compatível com OCI.

## Roteiro de Demonstração (Execução)

Para validar a integridade da arquitetura, o seguinte roteiro de execução deve ser seguido:

### 1. Preparação do Ambiente
```bash
# Iniciar banco de dados e pgAdmin
cd backend
docker-compose up -d

# Instalar dependências e preparar banco
npm install
npm run prisma:migrate:dev
npm run start:dev
```

### 2. Validação de Padrões e Fluxos
O sistema pode ser testado via Postman ou cURL. Abaixo, as principais evidências arquiteturais:

| Cenário | Padrão Evidenciado | Endpoint de Teste |
| :--- | :--- | :--- |
| **Cadastro de Usuário** | Chain of Responsibility | `POST /accounts/signup` (tente senhas curtas/sem números) |
| **Login e Sessão** | Factory / JWT | `POST /accounts/login` -> Recebe Token JWT |
| **Criação de Trilha** | Facade / RBAC | `POST /trilhas` (Requer role ORGANIZER) |
| **Cache de Consulta** | Cache Proxy | `GET /trilhas` (Segunda chamada é mais rápida/sem log de SQL) |
| **Finalização de Trilha** | Memento / Visitor | `POST /trilhas/:id/finalizar` |
| **Gerenciamento de Chat** | Object Pool | `GET /chat/pool/status` |

## Glossário Arquitetural

- **DTO (Data Transfer Object)**: Objetos simples usados para transferir dados entre as camadas de interface e aplicação.
- **Mapper**: Componente responsável por converter entidades de domínio em modelos de persistência (e vice-versa).
- **Vertical Slice**: Abordagem onde cada módulo contém todas as camadas necessárias para realizar uma funcionalidade completa.
- **RBAC (Role-Based Access Control)**: Controle de acesso baseado nos papéis atribuídos aos usuários.
- **IoC (Inversion of Control)**: Padrão onde o controle de execução é delegado a um framework (NestJS).

## Senso Crítico e Trade-offs

### Por que o Modelo 4+1?
O modelo 4+1 foi escolhido por ser o padrão de ouro na documentação acadêmica e profissional, permitindo que diferentes membros da equipe foquem em áreas específicas (ex: DBAs na Visão de Dados, Devs na Visão Lógica). Embora o **C4 Model** seja mais moderno e visual, o 4+1 oferece uma profundidade maior na descrição de processos e implementação que julgamos necessária para este projeto.

### Monólito vs Microsserviços
Optamos pelo monólito modular para reduzir a complexidade operacional. Em um cenário real de startup, a sobrecarga de gerenciar múltiplos deploys e comunicação entre serviços (latência, falhas parciais) não se justificaria para o volume inicial de usuários do BNB. A modularização rigorosa garante que o "caminho de volta" ou a extração para microsserviços seja trivial.
