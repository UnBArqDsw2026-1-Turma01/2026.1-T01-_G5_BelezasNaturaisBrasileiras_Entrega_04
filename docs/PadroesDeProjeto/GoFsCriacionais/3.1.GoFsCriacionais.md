# 3.1 Padrões de Projeto Criacionais (GoFs Criacionais)

## Introdução

Os Padrões de Projeto Criacionais (Creational Design Patterns) são um conjunto de soluções reutilizáveis que lidam com os mecanismos de criação de objetos, abstraindo a lógica de instanciação para tornar um sistema independente de como seus objetos são compostos e representados.

A importância dos padrões criacionais reside em que eles permitem que o código cliente não dependa das classes concretas dos objetos que precisam ser criados. Em vez disso, o cliente trabalha com abstrações (interfaces ou classes abstratas), o que reduz o acoplamento, aumenta a flexibilidade e facilita a extensão do sistema sem modificar o código existente.

## Padrões Criacionais

| Padrão               | Descrição                                                                                                             |     Status      | Onde Aplicar no Código                                                                                                                                                                                                                                                                                                                                                                    | Benefícios                                                                                                                                                               |
| :------------------- | :-------------------------------------------------------------------------------------------------------------------- | :-------------: | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Factory Method**   | Define uma interface para criar um objeto, permitindo que subclasses decidam qual classe instanciar.                  | ✅ Implementado | Criação de usuários por role (`CommonUserFactory`, `OrganizerUserFactory`, `AdminUserFactory`) e distribuição de diferentes tipos de `Badge` ao finalizar uma trilha (RF09, RN05).                                                                                                                                                                                                        | Escalabilidade para novos tipos de usuário e badge sem modificar o código cliente; centraliza lógica de criação.                                                         |
| **Singleton**        | Garante que uma classe tenha apenas uma instância e fornece um ponto de acesso global a ela.                          | ✅ Implementado | `ConfirmationCodeService` — garante uma única instância do serviço que gera, valida e revoga o código único de presença (RF06).                                                                                                                                                                                                                                                           | Evita inconsistências de estado ao gerir códigos de confirmação; reduz uso de memória.                                                                                   |
| **Prototype**        | Cria novos objetos clonando um objeto protótipo, em vez de criá-los do zero.                                          | ✅ Implementado | `User.clone()` via interface `IPrototype<User>`; duplicação de trilhas existentes para criar novas edições com mesmo ponto de encontro e estrutura base (RF08).                                                                                                                                                                                                                           | Cópias profundas e seguras sem efeitos colaterais; reduz erros de mutabilidade.                                                                                          |
| **Abstract Factory** | Fornece uma interface para criar famílias de objetos relacionados sem especificar suas classes concretas.             | ✅ Implementado | `UserFactoryRegistry` — registro central que mapeia cada `UserRole` para sua `IUserFactory` correspondente; o `CreateAccountUseCase` injeta `IUserFactoryRegistry` e chama `registry.get(role).create(...)` sem conhecer as factories concretas (RF01, RF02).                                                                                                                             | Garante consistência entre objetos de uma mesma família; elimina condicionais no código cliente; facilita adição de novos roles sem modificar o use case.                |
| **Builder**          | Constrói um objeto complexo passo a passo, separando a construção de sua representação.                               | ✅ Implementado | `UserBuilder` para montagem de usuários com campos opcionais; `TrilhaBuilder` para trilhas com muitos atributos opcionais (data, vagas, ponto de encontro, dificuldade) (RF03, RF08).                                                                                                                                                                                                     | Legibilidade na criação de objetos complexos; validação passo a passo; evita construtores com muitos parâmetros.                                                         |
| **Object Pool**      | Mantém um conjunto de objetos reutilizáveis pré-inicializados, entregando-os sob demanda e recuperando-os após o uso. | ✅ Implementado | `ChatObjectPoolService` (`IObjectPool<IChatConnection>`) gerencia até 50 conexões de chat reutilizáveis; `ChatSessionManagerService.withConnection(callback)` adquire e devolve conexaões automaticamente via try/finally; `ChatConnectionFactoryService` cria novas instâncias quando o pool está vazio. Exposto em `GET /chat/pool/status` e `POST /chat/sessions/:id/messages` (RF05). | Elimina o overhead de criar e destruir conexões por requisição; impõe limite explícito de 50 conexões simultâneas; desacopla o consumidor do ciclo de vida das conexões. |

> **Legenda de Status:** ✅ Implementado — já presente no código &nbsp;|&nbsp; ⚠️ Expansão de escopo — requer nova funcionalidade &nbsp;|&nbsp; ❌ Planejado — previsto para implementação no escopo atual
