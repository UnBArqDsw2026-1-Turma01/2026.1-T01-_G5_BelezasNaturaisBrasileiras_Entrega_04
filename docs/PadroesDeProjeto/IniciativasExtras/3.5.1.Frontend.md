## Introdução

Como iniciativa extra, a equipe desenvolveu uma aplicação web completa em React para consumir a API do Belezas Naturais Brasileiras. O frontend é uma **Single Page Application (SPA)** que expõe as principais funcionalidades do sistema para os três perfis de usuário: participante, organizador e administrador.

A interface foi construída com foco em simplicidade e funcionalidade, cobrindo os fluxos de autenticação, navegação de trilhas e pontos turísticos, gerenciamento de inscrições e chat — exercitando os mesmos padrões arquiteturais presentes no backend.

## Arquitetura

### Estrutura de Diretórios

```
frontend/src/
├── pages/          # Páginas da aplicação (uma por rota)
├── components/     # Componentes reutilizáveis (Navbar, ProtectedRoute)
├── contexts/       # Estado global — AuthContext
├── api/            # Módulos de chamadas ao backend (um por domínio)
├── hooks/          # Hooks customizados (jwtDecode)
├── App.tsx         # Definição de rotas e layout
└── main.tsx        # Entry point da aplicação
```

### Camada de API

Cada módulo em `src/api/` encapsula as chamadas a um domínio do backend, facilitando a manutenção e evitando acoplamento direto ao Axios nas páginas:

| Módulo          | Domínio                                           |
| :-------------- | :------------------------------------------------ |
| `auth.ts`       | Signup, login, promoção de usuário                |
| `axios.ts`      | Instância global com interceptor de JWT           |
| `trilhas.ts`    | CRUD de trilhas, finalizar, restaurar, badges     |
| `inscricoes.ts` | Solicitar, aceitar, rejeitar, check-in            |
| `pontos.ts`     | CRUD de pontos turísticos, finalizar              |
| `chat.ts`       | Sessões de chat e envio de mensagens              |
| `adapters.ts`   | Geocodificação, notificação, autenticação externa |

### Gerenciamento de Estado — `AuthContext`

O `AuthContext` gerencia a sessão do usuário em toda a aplicação:

- Persiste `token` e dados do `user` no `localStorage`
- Aplica **expiração automática de sessão em 3 horas** (`SESSION_DURATION_MS`)
- Expõe `login()`, `logout()`, `isAuthenticated` e o objeto `user` para os componentes
- O `ProtectedRoute` lê o contexto e redireciona para `/login` caso não autenticado ou sem o role exigido

## Controle de Acesso por Role

O `ProtectedRoute` implementa autorização baseada em roles no cliente:

```tsx
// Exemplo: rota acessível apenas por ORGANIZER ou ADMIN
<ProtectedRoute roles={["ORGANIZER", "ADMIN"]}>
  <Layout>
    <CriarTrilha />
  </Layout>
</ProtectedRoute>
```

| Role            | Acesso                                          |
| :-------------- | :---------------------------------------------- |
| Não autenticado | Páginas públicas apenas                         |
| `COMMON_USER`   | Páginas autenticadas (Perfil, Chat, CriarPonto) |
| `ORGANIZER`     | Tudo acima + CriarTrilha, PainelOrganizador     |
| `ADMIN`         | Tudo acima + PainelAdmin                        |

## Como Executar

### Pré-requisitos

- Node.js 20+
- Backend rodando em `http://localhost:3000`

### Instalação e execução

```bash
cd frontend
npm install
cp .env.example .env   # configure VITE_API_URL se necessário
npm run dev            # inicia em http://localhost:5173
```
