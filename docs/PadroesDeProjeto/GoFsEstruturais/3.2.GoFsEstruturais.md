# 3.2 Padrões de Projeto Estruturais (GoFs Estruturais)

## Introdução

Os Padrões de Projeto Estruturais (Structural Design Patterns) são soluções que lidam com a composição de classes e objetos para formar estruturas maiores, mantendo essas estruturas flexíveis e eficientes. Enquanto os padrões criacionais focam em como criar objetos, os padrões estruturais focam em como compor objetos e classes em estruturas maiores, reduzindo dependências e facilitando a manutenção.

Os padrões estruturais são essenciais para criar sistemas onde múltiplos componentes precisam trabalhar em conjunto de forma harmoniosa. Eles permitem que você combine objetos de diferentes formas sem acoplar o código cliente a implementações específicas, facilitando a extensão e a modificação do sistema sem breaking changes.


## Padrões Estruturais

| Padrão | Descrição | Status | Onde Aplicar no Código | Benefícios |
| :--- | :--- | :---: | :--- | :--- |
| **Adapter** | Converte a interface de uma classe em outra esperada pelos clientes, permitindo que classes incompatíveis trabalhem juntas. | ✅ Implementado | Módulo `adapters/` com três adaptadores independentes atrás de interfaces comuns: `GoogleMapsAdapter` (`IMapAdapter`) para geocodificação e rotas (`POST /adapters/geocode\|route`); `TwilioAdapter` (`INotificationAdapter`) para SMS e WhatsApp (`POST /adapters/notify/sms\|whatsapp`); `GoogleAuthAdapter` (`IAuthAdapter`) para validação de tokens externos (`POST /adapters/auth/validate`). Registros de uso persistidos em `GeolocationCacheRepository` e `NotificationLogRepository`. | Desacopla o sistema de fornecedores externos específicos; permite trocar provedores com mínima alteração no código cliente; isola dependências externas em um único módulo. |
| **Composite** | Compõe objetos em estruturas em árvore para representar hierarquias parte-todo, deixando os clientes tratarem objetos individuais e composições de forma uniforme. | ✅ Implementado | Hierarquia geográfica de Pontos Turísticos: Brasil → Região → Estado → Cidade → Ponto — suporta filtros hierárquicos (RF02) e cálculos recursivos como total de pontos por estado. | Modela hierarquias naturais do domínio; cliente trata folha e composto de forma uniforme; facilita operações recursivas. |
| **Decorator** | Adiciona responsabilidades a um objeto dinamicamente, fornecendo uma alternativa flexível ao uso de subclasses. | ✅ Implementado | `CachedTrilhaRepository` adiciona cache em memória e `AuditedTrilhaRepository` registra auditoria no `AuditLog`, ambos envolvendo o `PrismaTrilhaRepository` sem modificá-lo — composição `Audited → Cached → Prisma` via `useFactory` no módulo (RN01, RN07). | Adiciona responsabilidades em tempo de execução sem subclassing; responsabilidade única por decorator; facilita combinar comportamentos. |
| **Facade** | Fornece uma interface unificada e simplificada para um conjunto de interfaces em um subsistema. | ✅ Implementado | `InscricaoFacade` agrupa 5 Use Cases (SolicitarInscricao, Aceitar, Rejeitar, FazerCheckin, Listar); `TrilhaFacade` agrupa 3 Use Cases (Criar, Listar, Finalizar) — simplifica controllers de 5-7 dependências para 1. | Desacopla controllers dos Use Cases; reduz injeção de dependências; interface única e coesa para cada módulo. |
| **Proxy** | Fornece um substituto ou intermediário para outro objeto, controlando o acesso a ele. | ✅ Implementado | `TrilhaProxyRepository` intercepta `save()` e verifica via `TrilhaRequestContext` (AsyncLocalStorage) se o requisitante é o organizador antes de permitir escrita — autorização movida para fora do Use Case (RN02, RF08). | Controle de acesso transparente e centralizado; facilita auditoria sem alterar a lógica de negócio. |

> **Legenda de Status:** ✅ Implementado — já presente no código &nbsp;|&nbsp; ⚠️ Expansão de escopo — requer nova funcionalidade &nbsp;|&nbsp; ❌ Planejado — previsto para implementação no escopo atual
