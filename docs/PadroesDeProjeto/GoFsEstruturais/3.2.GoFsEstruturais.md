# 3.2 Padrões de Projeto Estruturais (GoFs Estruturais)

## Introdução

Os Padrões de Projeto Estruturais (Structural Design Patterns) são soluções que lidam com a composição de classes e objetos para formar estruturas maiores, mantendo essas estruturas flexíveis e eficientes. Enquanto os padrões criacionais focam em como criar objetos, os padrões estruturais focam em como compor objetos e classes em estruturas maiores, reduzindo dependências e facilitando a manutenção.

Diferente dos padrões criacionais, que focam no processo de criação, os padrões estruturais preocupam-se com a composição de classes ou objetos. Eles ajudam a garantir que, se uma parte de um sistema mudar, a estrutura inteira não precise ser alterada. No sistema Belezas Naturais Brasileiras, esses padrões são essenciais para integrar componentes heterogêneos, como SDKs de mapas, sistemas de arquivos e APIs de terceiros, de forma que o sistema permaneça coeso e modular.

## Padrões Estruturais

| Padrão        | Descrição                                                                                                                                                                         |        Status         | Onde Aplicar no Código                                                                                                                                         | Benefícios                                                                                                                                      |
| :------------ | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------: | :------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------- |
| **Adapter**   | Permite que objetos com interfaces incompatíveis colaborem entre si, atuando como um tradutor entre o sistema e serviços externos.                                                |    ✅ Implementado    | Módulo `adapters/` com adaptadores para `GoogleMaps`, `Twilio` e `GoogleAuth`, isolando a lógica de geocodificação, notificações e autenticação externa.       | Desacopla o sistema de fornecedores externos específicos; permite trocar provedores com mínima alteração no código cliente.                     |
| **Bridge**    | Divide uma classe grande ou um conjunto de classes intimamente ligadas em duas hierarquias separadas — abstração e implementação — que podem ser desenvolvidas independentemente. | ⚠️ Expansão de escopo | Separação entre tipo de notificação (abstrato) e canal de entrega (concreto: Email, SMS, Push) — requer implementação de sistema multi-canal.                  | Elimina a explosão combinatória de subclasses; permite que a abstração e a implementação variem de forma independente.                          |
| **Composite** | Permite compor objetos em estruturas de árvore e trabalhar com essas estruturas como se fossem objetos individuais.                                                               |    ✅ Implementado    | Hierarquia geográfica de Pontos Turísticos (Brasil → Região → Estado → Cidade → Ponto), permitindo tratar localizações individuais e grupos de forma uniforme. | Modela hierarquias naturais do domínio; facilita a implementação de operações recursivas em toda a árvore de localizações.                      |
| **Decorator** | Permite acoplar novos comportamentos a objetos ao colocá-los dentro de objetos invólucros especiais que contêm os comportamentos.                                                 |    ✅ Implementado    | `CachedTrilhaRepository` e `AuditedTrilhaRepository` envolvem o repositório base para adicionar funcionalidades de cache e auditoria de forma transparente.    | Adiciona responsabilidades em tempo de execução sem a necessidade de criar subclasses complexas; promove o Princípio da Responsabilidade Única. |
| **Facade**    | Fornece uma interface simplificada para uma biblioteca, um framework, ou qualquer outro conjunto complexo de classes.                                                             |    ✅ Implementado    | `InscricaoFacade` e `TrilhaFacade` simplificam o acesso a múltiplos Use Cases complexos a partir dos controladores da API.                                     | Reduz o acoplamento entre o cliente e o subsistema; simplifica a API exposta para os consumidores externos (como o frontend).                   |
| **Flyweight** | Permite ajustar mais objetos na quantidade disponível de RAM ao compartilhar partes comuns de estado entre múltiplos objetos em vez de manter todos os dados em cada objeto.      |    ✅ Implementado    | Compartilhamento de `Tag` entre trilhas e pontos turísticos para economizar memória em listagens extensas.                                                     | Reutiliza objetos imutáveis; reduz significativamente o consumo de memória quando há um grande volume de dados repetitivos.                     |
| **Proxy**     | Permite fornecer um substituto ou um espaço reservado para outro objeto para controlar o acesso a ele, agindo como um intermediário.                                              |    ✅ Implementado    | `TrilhaProxyRepository` intercepta chamadas de escrita para validar permissões de segurança antes de permitir que a operação chegue ao banco de dados.         | Controle de acesso transparente; permite a implementação de lógica auxiliar (segurança, cache, logging) sem alterar o objeto original.          |

> **Legenda de Status:** ✅ Implementado — já presente no código &nbsp;|&nbsp; ⚠️ Expansão de escopo — requer nova funcionalidade &nbsp;|&nbsp; ❌ Planejado — previsto para implementação no escopo atual
