# 3.2 Padrões de Projeto Estruturais (GoFs Estruturais)

## Introdução

Os Padrões de Projeto Estruturais (Structural Design Patterns) são soluções que lidam com a composição de classes e objetos para formar estruturas maiores, mantendo essas estruturas flexíveis e eficientes. Enquanto os padrões criacionais focam em como criar objetos, os padrões estruturais focam em como compor objetos e classes em estruturas maiores, reduzindo dependências e facilitando a manutenção.

Os padrões estruturais são essenciais para criar sistemas onde múltiplos componentes precisam trabalhar em conjunto de forma harmoniosa. Eles permitem que você combine objetos de diferentes formas sem acoplar o código cliente a implementações específicas, facilitando a extensão e a modificação do sistema sem breaking changes.


## Padrões Estruturais

| Padrão | Descrição | Status | Onde Aplicar no Código | Benefícios |
| :--- | :--- | :---: | :--- | :--- |
| **Adapter** | Converte a interface de uma classe em outra esperada pelos clientes, permitindo que classes incompatíveis trabalhem juntas. | ⚠️ Expansão de escopo | Adaptação de provedores de autenticação (`SupabaseAuthAdapter`, `GoogleOAuthAdapter`) a uma interface única `IAuthServiceAdapter` — requer adição de provedor alternativo (RF01); integração com API de mapas (Google Maps / Mapbox) para exibição de pontos turísticos. | Desacopla o sistema de fornecedores externos específicos; permite trocar provedores com mínima alteração no código. |
| **Bridge** | Desacopla uma abstração de sua implementação, permitindo que ambas variem independentemente. | ⚠️ Expansão de escopo | Separação entre tipo de notificação (abstrato: `NotificacaoParticipacao`, `NotificacaoAprovacao`) e canal de entrega (concreto: Email, SMS, Push, InApp) — requer implementação de sistema de notificações multi-canal. | Elimina a explosão combinatória de subclasses (N tipos × M canais); tipo e canal variam de forma independente. |
| **Composite** | Compõe objetos em estruturas em árvore para representar hierarquias parte-todo, deixando os clientes tratarem objetos individuais e composições de forma uniforme. | ❌ Planejado | Hierarquia geográfica de Pontos Turísticos: Brasil → Região → Estado → Cidade → Ponto — suporta filtros hierárquicos (RF02) e cálculos recursivos como total de pontos por estado. | Modela hierarquias naturais do domínio; cliente trata folha e composto de forma uniforme; facilita operações recursivas. |
| **Decorator** | Adiciona responsabilidades a um objeto dinamicamente, fornecendo uma alternativa flexível ao uso de subclasses. | ❌ Planejado | Repositórios decorados com cache (`CachedTrilhaRepository`) e auditoria (`AuditedTrilhaRepository`) sem modificar a implementação base — registra histórico de alterações de pontos turísticos (RN01, RN07). | Adiciona responsabilidades em tempo de execução sem subclassing; responsabilidade única por decorator; facilita combinar comportamentos. |
| **Facade** | Fornece uma interface unificada e simplificada para um conjunto de interfaces em um subsistema. | ❌ Planejado | `InscricaoFacade.inscreverEmTrilha()` orquestra: validar trilha + criar inscrição + abrir chat + notificar organizador (RF05, RN03); `TrilhaFacade.finalizar()` orquestra: inativar + buscar presentes + distribuir badges + notificar (RF09, RN05). | Simplifica chamadas complexas para o código cliente; reduz acoplamento com subsistemas internos. |
| **Flyweight** | Usa compartilhamento para sustentar eficientemente grandes quantidades de objetos finos, economizando memória. | ⚠️ Expansão de escopo | Compartilhamento de `Tag` entre trilhas e pontos turísticos via `TagFlyweightFactory` (ex: "Cachoeira", "Montanha", "Iniciante") — requer implementação de sistema de tags categorizáveis. | Reutiliza objetos imutáveis; reduz consumo de memória quando há grande volume de trilhas com as mesmas tags. |
| **Proxy** | Fornece um substituto ou intermediário para outro objeto, controlando o acesso a ele. | ❌ Planejado | `TrilhaProxyRepository` controla acesso ao repositório: verifica se o usuário é o organizador antes de permitir edição (RN02, RF08); registra audit log de acessos e alterações (RN01, RN07). | Controle de acesso transparente e centralizado; facilita auditoria sem alterar a lógica de negócio. |
