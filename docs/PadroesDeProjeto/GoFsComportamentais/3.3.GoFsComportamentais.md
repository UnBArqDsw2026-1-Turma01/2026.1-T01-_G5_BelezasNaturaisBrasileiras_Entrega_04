# 3.3 Padrões de Projeto Comportamentais (GoFs Comportamentais)

## Introdução

Os Padrões de Projeto Comportamentais (Behavioral Design Patterns) lidam com a comunicação entre objetos e com a distribuição de responsabilidades. Enquanto os padrões criacionais focam em criar objetos e os estruturais em compor estruturas, os comportamentais focam em como os objetos interagem e colaboram, bem como em como distribuem responsabilidades de forma eficiente.

Os padrões comportamentais são especialmente úteis em sistemas complexos onde múltiplos componentes precisam se comunicar, reagir a mudanças de estado, iterar sobre coleções ou processar comandos de forma padronizada. Eles melhoram a legibilidade, manutenibilidade e extensibilidade do código, facilitando o entendimento do fluxo de execução e das responsabilidades de cada classe.


## Padrões Comportamentais

| Padrão | Descrição | Status | Onde Aplicar no Código | Benefícios |
| :--- | :--- | :---: | :--- | :--- |
| **Chain of Responsibility** | Passa um pedido ao longo de uma cadeia de handlers, onde cada um decide se processa o pedido ou o passa adiante. | ❌ Planejado | Cadeia de autorização por role: `AdminAuthHandler` → `OrganizerAuthHandler` → `CommonUserAuthHandler` → `VisitorAuthHandler`; validação em cascata no cadastro (email → senha → termos) (RF01, RN02). | Desacopla quem solicita de quem processa; permite adicionar ou reordenar handlers sem alterar o código cliente. |
| **Command** | Encapsula um pedido como um objeto, permitindo parametrizar clientes com diferentes pedidos, enfileirar, registrar e executar operações. | ❌ Planejado | `EditarPontoTuristicoCommand` encapsula cada edição como objeto com `execute()` e `undo()`, formando o histórico de versões (RN01); fila de tarefas em background para envio de badges e e-mails (RF09, RN05). | Permite enfileirar, registrar e desfazer operações; implementa histórico de edições de forma elegante. |
| **Interpreter** | Define uma representação gramatical para uma linguagem e um interpretador para processar sentenças dessa linguagem. | ⚠️ Expansão de escopo | DSL de busca avançada de trilhas: `"estado:CE AND tipo:Cachoeira AND dificuldade:Média"` — requer implementação de filtros customizados; ou regras dinâmicas para conquista de badges (ex: `"trilhas_completas >= 10 AND km_total > 50"`). | Empodera usuários com filtros expressivos; separa sintaxe de semântica; facilita adição de novas regras sem recompilar. |
| **Iterator** | Fornece uma forma de acessar sequencialmente os elementos de uma coleção sem expor sua representação subjacente. | ❌ Planejado | `TrilhaIterator` abstrai paginação do feed de trilhas (RF02); iteração lazy-load de comentários em pontos turísticos (RF04); iteração de badges de um usuário com filtros (RF09). | Abstrai detalhes de paginação e lazy-loading; reutilizável em múltiplas listagens do sistema. |
| **Mediator** | Define um objeto que encapsula como um conjunto de objetos interage, promovendo acoplamento fraco. | ❌ Planejado | `ChatMediator` garante a regra crítica de comunicação estritamente 1-para-1 entre participante e organizador: valida que o chat tem exatamente 2 participantes, controla permissões de envio e verifica a existência de inscrição válida (RN03, RF05). | Centraliza e garante a regra de negócio RN03; reduz acoplamento direto entre os objetos que se comunicam. |
| **Memento** | Captura e externaliza um estado interno de um objeto sem violar o encapsulamento, permitindo que o objeto seja restaurado para esse estado mais tarde. | ❌ Planejado | `PontoTuristicoMemento` registra snapshot do estado a cada edição, incluindo autor e timestamp (RN01); snapshot de `Trilha` antes da exclusão, preservando dados de badges e comentários (RN07). | Implementa histórico e auditoria sem violar encapsulamento; suporta restauração de estados anteriores. |
| **Observer** | Define uma dependência um-para-muitos entre objetos, de modo que quando um objeto muda de estado, todos os seus dependentes são notificados automaticamente. | ✅ Implementado | `TrilhaEventEmitter` notifica `BadgeDistributorObserver` ao finalizar trilha (RF09, RN05); expansão para notificar admin em denúncias abertas (RF12), usuário ao receber avaliação (RN04) e ao editar local (RN01). | Desacopla o gerador de eventos de seus listeners; novos comportamentos são adicionados sem alterar o código emissor. |
| **State** | Permite que um objeto altere seu comportamento quando seu estado interno muda, parecendo mudar a classe do objeto. | ✅ Implementado | Máquina de estado de `Trilha`: `TrilhaAtivaState` → `TrilhaLotadaState` → `TrilhaInativaState`, com retorno automático a ativo se data mudar (RN06); máquina de estado de `Inscricao`: PENDENTE → ACEITA/REJEITADA → PRESENTE (RF07). | Encapsula lógica de transição de estado; elimina cascatas de `if/else`; valida transições ilegítimas em tempo de execução. |
| **Strategy** | Define uma família de algoritmos, encapsula cada um e torna-os intercambiáveis, deixando o cliente escolher qual usar. | ❌ Planejado | Estratégias de validação de código de confirmação (`SimpleCodeStrategy`, `RateLimitedCodeStrategy` — máx 3 tentativas/hora) (RF06); estratégias de filtro de trilhas por geografia, tempo e dificuldade (RF02). | Permite trocar algoritmos em runtime sem alterar o código que os usa; elimina cascatas de condicionais. |
| **Template Method** | Define o esqueleto de um algoritmo em uma operação de classe base, deixando que subclasses preencham os passos específicos. | ❌ Planejado | `ModeracaoTemplate` define o fluxo invariante: abrir denúncia → coletar evidências → tomar decisão → executar → notificar; subclasses `ModeracaoConteudo`, `ModeracaoUsuario`, `ModeracaoTrilha` implementam as etapas específicas (RF12, RN02). | Evita duplicação de passos fixos; garante que o processo de moderação siga sempre a mesma estrutura. |
| **Visitor** | Representa uma operação a ser realizada nos elementos de uma estrutura de objetos, deixando que você defina uma nova operação sem mudar as classes dos elementos. | ⚠️ Expansão de escopo | `ExportPDFVisitor`, `ExportJSONVisitor`, `ExportCSVVisitor` para exportar dados de trilhas e inscrições em múltiplos formatos — requer implementação de funcionalidade de exportação/relatórios para organizadores e admins. | Adiciona novas operações sobre a estrutura de dados sem modificar as classes de domínio; separa algoritmo da estrutura. |

> **Legenda de Status:** ✅ Implementado — já presente no código &nbsp;|&nbsp; ⚠️ Expansão de escopo — requer nova funcionalidade &nbsp;|&nbsp; ❌ Planejado — previsto para implementação no escopo atual
