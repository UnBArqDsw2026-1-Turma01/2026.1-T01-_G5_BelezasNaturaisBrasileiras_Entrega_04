# 3.3 Padrões de Projeto Comportamentais (GoFs Comportamentais)

## Introdução

Os Padrões de Projeto Comportamentais (Behavioral Design Patterns) lidam com a comunicação entre objetos e com a distribuição de responsabilidades. Enquanto os padrões criacionais focam em criar objetos e os estruturais em compor estruturas, os comportamentais focam em como os objetos interagem e colaboram, bem como em como distribuem responsabilidades de forma eficiente.

Os padrões comportamentais são especialmente úteis em sistemas complexos onde múltiplos componentes precisam se comunicar, reagir a mudanças de estado, iterar sobre coleções ou processar comandos de forma padronizada. Eles melhoram a legibilidade, manutenibilidade e extensibilidade do código, facilitando o entendimento do fluxo de execução e das responsabilidades de cada classe.


## Padrões Comportamentais

| Padrão | Descrição | Onde Aplicar no Código | Benefícios |
| :--- | :--- | :--- | :--- |
| **Chain of Responsibility** | Passa um pedido ao longo de uma cadeia de handlers, onde cada um decide se processa o pedido ou o passa adiante. | | |
| **Command** | Encapsula um pedido como um objeto, permitindo parametrizar clientes com diferentes pedidos, enfileirar, registrar e executar operações. | | |
| **Interpreter** | Define uma representação gramatical para uma linguagem e um interpretador para processar sentenças dessa linguagem. | | |
| **Iterator** | Fornece uma forma de acessar sequencialmente os elementos de uma coleção sem expor sua representação subjacente. | | |
| **Mediator** | Define um objeto que encapsula como um conjunto de objetos interage, promovendo acoplamento fraco. | | |
| **Memento** | Captura e externaliza um estado interno de um objeto sem violar o encapsulamento, permitindo que o objeto seja restaurado para esse estado mais tarde. | | |
| **Observer** | Define uma dependência um-para-muitos entre objetos, de modo que quando um objeto muda de estado, todos os seus dependentes são notificados automaticamente. | | |
| **State** | Permite que um objeto altere seu comportamento quando seu estado interno muda, parecendo mudar a classe do objeto. | | |
| **Strategy** | Define uma família de algoritmos, encapsula cada um e torna-os intercambiáveis, deixando o cliente escolher qual usar. | | |
| **Template Method** | Define o esqueleto de um algoritmo em uma operação de classe base, deixando que subclasses preencham os passos específicos. | | |
| **Visitor** | Representa uma operação a ser realizada nos elementos de uma estrutura de objetos, deixando que você defina uma nova operação sem mudar as classes dos elementos. | | |
